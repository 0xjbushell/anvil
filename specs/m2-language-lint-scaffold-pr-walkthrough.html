Mental model

The manifest is the contract between static files and generated projects

Anvil scaffolding already had seed code and project templates. This PR extends that contract with language-specific lint tool directories. The key review question is: for every file under static/<lang>/..., does src/manifest.ts point to the same destination a generated project should receive?

Why the manifest changes matter

The scaffold engine copies by destination, not by discovery. If the static tree and manifest path drift apart, tests can pass by importing static files directly while generated projects silently miss those files. This PR adds tests around those edges.

Language scaffolds

Each language gets an empty but runnable rule host

The files are deliberately boring: small loaders, empty rule maps, stub checker functions, and harness tests. That is the point. Future tickets can add rule logic without also inventing packaging, import style, or generated-project layout.

TypeScript / JavaScript

Adds a CommonJS ESLint plugin scaffold at static/typescript/tools/lint-rules/plugin.js.

  • rules = {} keeps the plugin valid before rules exist.
  • configs.recommended is a flat config with a self-reference.
  • package.json sets "type": "commonjs" near the plugin.
  • helpers.js creates a future-ready ESLint v9 RuleTester.

Go

Adds an empty multichecker binary at static/golang/tools/go-analyzers/cmd/anvil-lint/main.go.

  • multichecker.Main() starts with no analyzers registered.
  • go.mod.ejs renders go <%= toolchain.go %>.
  • Makefile builds bin/anvil-lint from the command package.
  • testdata/.gitkeep reserves space for future analyzer fixtures.

Python

Adds a Flake8 plugin package under static/python/tools/flake8-plugin/anvil_lint.

  • AnvilChecker exposes Flake8 name, version, and run().
  • Stub modules cover anti-slop, error-handling, structural, and test-quality categories.
  • setup.py and setup.cfg register ANV = anvil_lint:AnvilChecker.
  • Generated-project tests exercise checker loading and empty-result behavior.

Manifest wiring

The destination paths are what generated projects see

A reviewer should read src/manifest.ts as the source of truth for what a generated project receives. The static source path is derived from the language and destination, except for the Go analyzer go.mod, which is a static EJS template rendered through staticTemplateEntry().

Review findings addressed

Three subtle scaffold-contract issues were fixed before the PR opened

These are useful places for reviewers to focus because they are not obvious from looking at a single file. Each issue was about keeping generated-project behavior aligned with the static tree and test expectations.

1
Go testdata now copies into generated projects The static testdata/.gitkeep directory was present and tested, but generated projects would miss it until tools/go-analyzers/testdata/**/* was added to the Go manifest.
2
TypeScript static paths now match manifest destinations Static files moved under static/typescript/tools/lint-rules/** so staticEntry("typescript", "tools/lint-rules/...") resolves to real files.
3
Python includes the error-handling checker category The Flake8 scaffold now has error_handling.py, imports it, delegates from AnvilChecker.run(), and covers it in scaffold tests.

Validation story

The PR proves scaffold shape, generated compatibility, and repository health

The validation set intentionally mixes broad repo gates with focused scaffold checks. Broad gates catch regressions in existing Anvil behavior; focused checks prove the new language tool trees are wired and runnable enough for future rule tickets.

How to think about follow-up work

Future rule tickets should add one rule at a time inside these category folders, then extend the matching test harness. They should not need to redesign plugin packaging, analyzer module structure, or manifest destination paths.
