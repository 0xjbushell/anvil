System overview

The PR builds the load-bearing scaffold layer, then wraps it in a local test loop

Think of M1 as two related tracks. The first track defines how anvil will decide what files belong in a scaffolded project and how to write them safely. The second track gives developers a repeatable sandbox so they can exercise those decisions without touching real projects.

What changed in one sentence

M1 does not finish the end-user anvil init command; it creates the reusable machinery that init will need: manifest-driven file generation, safe re-scaffold semantics, scenario fixtures, targeted agent checks, and local/CI gates.

Production path

How the scaffold engine writes files safely

The scaffold engine is centered on scaffold(ctx, options). It takes a ScaffoldContext, builds an in-memory tree from the manifest, compares that tree to disk, resolves conflicts, writes a checkpoint lockfile, writes files one at a time, and finalizes the lockfile.

Shared vocabulary

src/types.ts defines the common nouns: Lang, ScaffoldContext, ManifestEntry, FsTreeEntry, conflict types, and AnvilLockfile. Start here when a type appears unfamiliar.

  • ScaffoldContext records project name, language, target dir, non-interactive mode, toolchain, and deterministic year.
  • ManifestEntry can be static, template, or generator.
  • LockfileEntry.status is written or pending for checkpoint recovery.

Manifest-driven output

src/manifest.ts builds per-language lists for TypeScript, Go, and Python. It has helper constructors for static files, EJS templates, and generator slots. Seed files are conditional so existing projects can avoid unwanted starter code.

  • staticEntry() maps from static/<lang>/... to target paths.
  • templateEntry() maps from src/templates/<lang>/...ejs.
  • generatorEntry() is declared but intentionally unsupported by the engine in M1.

In-memory FsTree

src/scaffold/fstree.ts stages generated content before disk writes. It rejects absolute or escaping paths, compares staged content to disk, and flushes only create or update changes.

  • Path normalization uses POSIX relative paths so manifests behave consistently.
  • listChanges() classifies each entry as create, update, or unchanged.
  • flushChanges() creates parent directories and writes UTF-8 text.

Lockfile checkpointing

src/scaffold/lockfile.ts owns .anvil.lock. The lockfile is JSON, uses sha256:<hex> checksums, records scaffold context/toolchain, and uses flushStatus to distinguish complete runs from interrupted runs.

  • readLockfile() returns absent, complete, in-progress, or corrupt.
  • Text checksums normalize line endings; known binary extensions preserve bytes.
  • Atomic writes reduce the chance of half-written lockfiles.

Conflict boundaries

src/scaffold/engine.ts requires exactly one conflict boundary: interactive onConflict or non-interactive onReport. This keeps write behavior explicit.

  • Interactive mode asks per update, then writes only approved changes.
  • Non-interactive mode reports all updates and throws before writing anything.
  • Abort produces zero data writes and no checkpoint lockfile.

Directory lock guard

src/scaffold/dirlock.ts prevents concurrent scaffold runs against the same target. It uses a vendored lock module plus a .anvil.lock.pid payload containing PID and process start time.

  • Live locks fail with a clear PID message.
  • Dead or mismatched PID payloads can be reclaimed as stale.
  • Missing or unreadable fresh payloads are treated as possibly live, not overwritten.
scaffold(ctx, options)
  validate exactly one conflict boundary
  read .anvil.lock and reject corrupt/cross-language states
  build FsTree from language manifest
  compare tree against targetDir
  resolve update conflicts
  write in-progress checkpoint lockfile
  flush each data change
  mark each lockfile entry written
  finalize flushStatus = "complete"
Junior-reviewer tip

Pay attention to where writes happen. The engine intentionally separates "decide what should change" from "write to disk." Most safety properties come from that separation.

Developer inner loop

The sandbox harness makes scaffold behavior reviewable without mutating real projects

The dev harness adds a small scenario language and three commands: bun dev for manual debugging, bun fixtures for the full regression catalog, and bun agent:check for targeted checks based on changed files.

Scenario schema

src/dev/schema.ts uses Zod to parse YAML. A scenario must specify exactly one driver: plain args or future pty. The schema is strict, so misspelled assertion keys fail instead of being silently ignored.

  • Input names must be directory names, not paths.
  • Assertions include exit code, file existence, content, regex, stdout/stderr, and unchanged-tree checks.
  • PTY shape is parsed now, but execution is intentionally blocked until the init integration exists.

Scenario runner

src/dev/harness.ts loads a scenario, copies its input fixture into a temporary workdir, runs bun bin/anvil.ts ..., evaluates assertions, deletes successful workdirs, and preserves failing workdirs for debugging.

  • Missing inputs are internal errors, not test failures.
  • Multiple assertion failures are collected together.
  • Passing scenarios clean up after themselves.

bun dev

src/dev/cli.ts exposes bun dev <scenario> for manual exploration. It copies an input fixture into .sandbox/scratch by default, prints the absolute path, and prints a ready-to-run command hint on stderr.

  • Default runs wipe and recopy the scratch directory.
  • --keep overlays inputs but rejects symlinked targets before copying.
  • --name <dirname> lets reviewers create named sandboxes under .sandbox/.

bun fixtures

The fixtures command discovers committed YAML scenarios, runs them sequentially, prints per-scenario pass/fail lines, supports --filter, and returns non-zero when any scenario fails.

  • Failures include assertion messages and the preserved workdir path.
  • A filter with zero matches is an error, not a false green.
  • The package script is bun src/dev/cli.ts fixtures.
Targeted checks

bun agent:check chooses the smallest useful fixture set

The targeted runner is meant for agents and contributors making small changes. It reads changed files from git diff --name-only HEAD, maps those paths to relevant scenarios, and stays quiet on green.

Subtle but important regression guard

Current TypeScript re-scaffold fixture names do not contain "typescript" or "ts". The selection logic enriches discovered scenarios with .anvil.lock language metadata from their input fixture, so a TypeScript template change selects re-scaffold-clean, re-scaffold-drift, and re-scaffold-template-bumped instead of falsely running nothing.

How to review

A suggested reading order for junior reviewers

The PR is easier to understand if you read it from contracts outward, not alphabetically. This order builds the mental model before the tests.

1. Start with nouns Read src/types.ts and src/manifest.ts. Ask: "What state does the system need to remember?"
2. Read the engine path Read fstree.ts, lockfile.ts, then engine.ts. Ask: "What happens before disk writes?"
3. Review failure behavior Read conflict, checkpoint, and dirlock tests. Ask: "If a run fails halfway, can the next run understand it?"
4. Read the harness Read schema.ts, harness.ts, and assertions.ts. Ask: "Can failures be diagnosed quickly?"
5. Read dev CLI behavior Read src/dev/cli.ts and changed.ts. Ask: "Do commands fail closed rather than greenwashing?"
6. Finish with gates Read hook, workflow, README, and AGENTS tests. Ask: "Will future contributors run the right checks?"

Deferral: TIX-000023

Full anvil init integration is deferred because the current manifest points at language static/template/generator payloads that are not fully present yet. The engine also intentionally throws for generator entries. Implementing init in this PR would either pull later language-vertical work into M1 or require shortcuts around the architecture.

Deferral: TIX-000064

PTY interactive scenarios depend on real interactive anvil init prompts and generated output. The schema already models PTY scripts, but harness.ts rejects them with a clear unsupported message until TIX-000023 lands.

Most important reviewer question

Does every potentially destructive path fail safely? In this PR, that means non-interactive conflicts should write nothing, interrupted flushes should leave understandable checkpoint state, sandbox copies should not follow unsafe paths, and targeted checks should avoid false greens.

Closeout

What this PR enables next

Once reviewed and merged, the next milestone can build on stable primitives instead of inventing them: language assets can be added to the manifest, generator support can be implemented deliberately, anvil init can call the engine, and the fixture catalog can graduate from CLI smoke checks to real scaffold acceptance tests.

For product behavior

The scaffold engine can become the backend of anvil init once language payloads exist. The existing checkpoint and conflict semantics mean init does not need to invent write safety from scratch.

For contributors

Contributors now have a local loop: use bun dev to inspect a sandbox, bun agent:check for focused validation, and bun fixtures for the full gate.

For agents

Root AGENTS.md points agents to the project shape, decision file, fixture commands, and reference-implementation discipline, reducing repeated onboarding mistakes.
