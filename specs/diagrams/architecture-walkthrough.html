Architecture Walkthrough · For new contributors

anvil — a CLI scaffolder built to fight agentic slop

anvil is a single-command tool that generates a complete, opinionated project skeleton (TypeScript, Go, or Python) wired with custom lint rules, a quality toolchain, git hooks, and an AGENTS.md file. Its purpose is narrow and explicit: make it harder for AI coding agents to produce structurally bad code in the first place, not detect it after the fact.

01 The problem we're solving, and the shape of the solution

If you remember one thing from this doc, remember this section.

Coding agents (Claude, GPT, Copilot) routinely produce code that passes basic linting but is bad code. They generate empty catch blocks that swallow errors. They split implementations into 30 microscopic wrapper files. They write tests that expect(true).toBe(true). They paste // TODO: implement later and move on. None of this is caught by ESLint-recommended, golangci-lint defaults, or Ruff defaults — those tools focus on syntax, not engineering judgment.

anvil's bet is that the cheapest place to prevent slop is at project birth — by generating a project where the rules, conventions, and seed code already model what "good" looks like. Once an agent is dropped into a project with strict guardrails plus an AGENTS.md that explains them, it tends to comply. The three pillars:

Pillar 1 · Seed code

Real working code in the conventional file layout (seed/). Agents mimic what they see — give them good shapes to mimic.

Pillar 2 · Custom lint rules

9 anti-slop + 10 structural + 5 test-quality rules implemented natively in each language's lint ecosystem (ESLint plugin, go vet -vettool, Flake8 plugin).

Pillar 3 · Pre-commit + pre-push hooks

Failures block commits. There is no "fix it later" — the loop closes locally before bad code ever leaves the developer's machine. CI is explicitly not in scope (D-38).

Why "local-first"? anvil owns the dev environment, not deployment. We don't ship CI workflows because every team's CI looks different — but make check is "CI-ready by design": it's the same command a CI job would run.

02 System architecture — the big picture

Five major component families. Two of them are shared (yellow boxes); three are language-specific and live behind a common interface.

Why a plugin model for languages? Each language has fundamentally different lint tooling (ESLint plugin vs Go vettool vs Flake8 plugin). Hiding all three behind a common LanguagePlugin interface lets the scaffold engine stay language-agnostic: it asks "what files do you want me to write?" and "what context do they need?" — the plugin answers, and the engine handles the FS work.

Why two lockfiles? .anvil.lock is the durable record of "what anvil scaffolded last time" — read on re-runs. .anvil.lock.pid is a short-lived concurrency guard that exists only while a scaffold is in progress. It records { pid, startTime } so a stale lock from a crashed process is safely reclaimed even if the OS reused the PID (D-59).

Five types do most of the work. They're defined once in the engine and imported by every plugin. Don't duplicate them.

FsTree read(path) · write(path, content) · exists(path) · delete(path) · rename(old, new) · entries() · listChanges() · flushChanges() The central abstraction. An in-memory virtual filesystem. The engine writes everything here first, classifies CREATE/UPDATE/SKIP against disk, prompts the user, then flushes to disk. See § 5.
FsTreeChange { path, action: 'CREATE'|'UPDATE'|'SKIP', oldContent?, newContent? } What listChanges() returns — the diff between FsTree and disk. Drives conflict prompts.
AnvilLockfile { version, lang, context, files: LockfileEntry[], createdAt, updatedAt }
LockfileEntry = { path, checksum, source }		 Written to .anvil.lock after a successful scaffold. Lets future runs detect what anvil owns vs what the user has modified.
ConflictAction 'overwrite' | 'skip' | 'abort'  (3 values, no 'diff') The terminal result of a conflict prompt. 'diff' is internal prompt-loop state — never returned.
Why does the engine restrict file content to strings (no Buffers)? Because v1 only generates text files (source code, configs, markdown). Supporting Buffers would force every plugin to handle two encodings forever, for zero v1 benefit. If we ever need to scaffold binaries, that's a v2 decision.

05 FsTree — the heart of the scaffold engine

If you only learn one component, learn this one. Every other engine module exists to feed it or read from it.

FsTree is a two-pass file writer. Pass 1: render every file into a Map<path, content> in memory. Pass 2: walk that map, compare each entry to whatever is on disk, classify the difference, ask the user about conflicts, then commit. Only after Pass 2 succeeds does anything reach disk.

Why "2-way" not "3-way" diff in v1? A 3-way merge would also re-render the original template and try to detect "user modified vs upstream template changed." That's what tools like yo and cookiecutter attempt — and they do it badly. anvil update is deferred to v2 (D-39). For v1, FsTree only asks "what's in memory vs what's on disk?" and prompts on every UPDATE. The lockfile stores checksums but the prompt-gating logic doesn't consult them.
Why is flush sequential, not transactional? Atomic FS writes across N files would require a journal or staging directory. The cost of a partial-write failure in v1 is "user re-runs anvil init" — totally fine. We chose simplicity.

06 The lint rule pipeline (a separate concern)

The scaffold engine builds a project. The lint rules guard it. They share no code — they don't need to.

Once a project is scaffolded, the only anvil code that runs in it is the per-language lint plugin (ESLint plugin / Go vettool binary / Flake8 plugin). The plugins are distributed as source files copied into the project (not installed from a registry — D-15..D-17). This means the rules ship with the project, are version-pinned by the scaffold, and can be modified by the team without touching anvil.

07 How anvil itself ships

No npm. No Docker. Bun-native binary, period.

anvil is a Bun + TypeScript project (D-45). It ships two ways:

  • bunx anvil init — for users who already have Bun installed. Bun runs the TypeScript directly; no build step.
  • Compiled standalone binary — bun build --compile produces self-contained binaries for 5 targets (linux x64/arm64, darwin x64/arm64, windows x64). Distributed via a curl install.sh script. No Bun, Node, or npm required on the user's machine.
Why no npm distribution? Two reasons. (1) anvil's audience is agents and engineers who work across multiple language stacks — npm-only would alienate Go/Python devs. (2) Bun's --compile output is a single file with no runtime dependency, which is the right shape for a CLI tool. Adding an npm package later is a v2 decision (D-45).

If you're just starting

Read in this order: PROJECT.md → REQUIREMENTS.md → scaffold-engine.md §"FsTree Virtual File System" → lint-rules.md Category A overview. Then run tix list --todo and pick a leaf deliverable — they're sized for one PR and have explicit acceptance criteria.

Convention reminder: never edit .tix/ JSON files directly. Always go through tix update, tix start, tix complete. The state machine is enforced; bypassing it corrupts the audit trail.
