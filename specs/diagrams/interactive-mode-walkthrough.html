anvil · CLI UX walkthrough

Interactive Mode

How anvil init talks to humans, and what changes when no human is on the other end. Walks through TTY detection, the two prompt phases (setup & conflict), the per-file conflict loop, and the non-interactive defaults that make agents and CI work safely.

01The premise

Every anvil init run needs answers to two kinds of questions:

Phase A

Setup questions (asked once, upfront)

  • What language? --lang
  • What project name?
  • Which package manager? (TS only)
  • Generate seed code, or skip it because the repo already has app code?

One round of questions, then anvil starts working.

Phase B

Conflict questions (asked per file, during flush)

  • This file already exists on disk and anvil wants to write a different version. What now?
  • Choices: overwrite, skip, diff (preview), abort
  • Asked only for files classified UPDATE. CREATE files just get written.

Zero on a fresh greenfield. Many on a re-scaffold.

Why split it this way?

Setup answers are about what to build. Conflict answers are about how to handle disk reality. Mixing them would mean re-prompting forever as anvil discovers files mid-render. The FsTree lets us batch all conflict decisions after rendering is complete — so you see the full picture before approving any change.

02Mode selection — explicit, never inferred

Anvil decides interactive vs non-interactive based on one signal only: the --non-interactive flag. This is opt-in. Piping into anvil without the flag is treated as a user error, not a silent mode switch (D-67 supersedes D-56).

What about piping?

If you run echo y | anvil init without --non-interactive, anvil errors immediately rather than silently switching modes. The previous auto-detect behavior (D-56) caused a real failure mode: piped runs would silently skip every conflict and print "success" while applying nothing. Forcing the flag makes intent explicit.

03Phase A — Setup prompts

Powered by @inquirer/prompts (D-23). Each question has a resolver chain — anvil tries to answer it without asking, and only prompts if every source comes up empty.

Resolver order (D-67 detection-first)

OrderSourceExample
1Explicit CLI flag--name myapp wins, never prompted
2Detected from diskbun.lock exists → packageManager = bun
3Existing .anvil.lockRe-scaffold reads previous answers

5Safe defaultNon-interactive fallback (e.g. main for branch, dir basename for name)

04Phase B — Per-file conflict prompts

Triggered only for files the FsTree classifies as UPDATE. Each conflict is a small loop — the user can preview a diff as many times as they want before committing to a terminal action.

The state machine

What it looks like — re-scaffold with two conflicts

~/code/myapp — conflict resolution
⚠ 2 files have local changes that differ from anvil's templates. [1/2] Makefile (modified by you, anvil has updates) ? What should I do? ❯ skip — keep your version (default) overwrite — take anvil's version diff — show me the changes first abort — stop scaffold, write nothing › diff ──────────────── Makefile ──────────────── - typecheck: - tsc --noEmit + typecheck: + tsc --noEmit --strict + @echo "✓ typecheck passed" ──────────────────────────────────────────── ? What should I do? (prompt repeats) ❯ skip overwrite diff abort › overwrite ✔ [2/2] .eslintrc.json (modified by you, anvil has updates) ? What should I do? ❯ skip ✔ ⚒ writing to disk… ✔ Done. 1 updated · 1 skipped · 19 unchanged. Skipped: .eslintrc.json (kept your version)
Why re-prompt after diff?

Diff is a preview, not a decision. Letting it loop means you can inspect a change carefully, decide it's fine, and accept — or scroll back, look again, and skip. Cruft and Yeoman both make you commit before seeing the diff; we won't.

Abort is atomic

If you pick abort on file 47 of 50, the 46 already-approved changes are not written. No .anvil.lock update, no partial state, no git status surprise. The flush phase only runs after every conflict has a non-abort answer.

05Non-interactive equivalence

When you pass --non-interactive, every prompt above must answer itself. Setup prompts use safe defaults; conflicts are not auto-resolved — they're reported and the run aborts with no writes. Mapping per D-67:

PromptInteractive answerNon-interactive resolution
projectName user types or accepts default directory basename of target path
packageManager detect → pick from list if no lockfile detect → fall back to bun
defaultBranch user types or accepts main main
generateSeed Y/n auto: false if hasCode heuristic fires, else true
UPDATE conflicts per-file overwrite/skip/diff/abort diff report on stderr → exit non-zero, no writes
Exit code 0 (success) or 1 (abort) 0 (no conflicts) or non-zero (conflicts present)

The conflict reporter — anvil reports, the agent decides

When --non-interactive finds files that would be UPDATEs (template differs from disk), anvil refuses to choose for you. Instead it prints a unified-diff report, exits non-zero, and writes nothing. The agent (or human running headless) reads the report, decides what to do, and re-runs. This is the same mental model as a compiler error: surface the problem, let the caller fix it.

~/code/myapp — agent inner loop
$ anvil init --non-interactive ⚒ anvil v0.1.2 · re-scaffolding · non-interactive ⚒ 21 files queued · 18 unchanged · 3 conflicts ✗ 3 conflicts — anvil's templates differ from your local files. No files written. ──── Makefile ──── - typecheck: - tsc --noEmit + typecheck: + tsc --noEmit --strict + @echo "✓ typecheck passed" ──── .eslintrc.json ──── - "no-console": "warn" + "no-console": "error" ──── src/seed/index.ts ──── [... unified diff ...] To resolve, for each conflict: • edit the file to match anvil's version (then re-run) • rm the file (anvil will recreate it next run) • leave it (next run will report the same conflict) $ echo $? 1
edit to match

Take anvil's version. Agent reads the diff, applies it (or fully replaces the file). Next run sees no conflict.

delete

Start over. rm the file, anvil treats it as CREATE on the next run. Useful when the local version is junk.

leave it

Live with the noise. Conflict re-appears every run — honest signal that anvil and disk disagree. No silent acceptance.

Why no --on-conflict flag?

Earlier sketches included --on-conflict skip|overwrite|backup. Dropped. A "force" knob re-introduces the silent-skip and silent-overwrite failure modes the conflict reporter exists to eliminate. The agent has cat, diff, git, and a brain — that's a richer toolkit than any flag could be.

06End-to-end — the whole flow on one page

  1. Parse args. Read --lang, --non-interactive, --dry-run, etc.
  2. Set mode. If --non-interactive flag present → non-interactive. Otherwise interactive (a TTY is assumed and prompts will fail visibly without one).
  3. Detect disk state. Look for .anvil.lock, lockfiles, existing source code (hasCode heuristic).
  4. Resolve setup answers using the chain: flag → detected → lockfile → prompt (interactive only) → safe default (non-interactive).
  5. Acquire .anvil.lock.pid to prevent concurrent scaffold (D-57b).
  6. Render FsTree. Templates rendered into in-memory virtual filesystem. No disk writes yet.
  7. Classify each path as CREATE, UPDATE, or no-op (checksum match).
  8. Resolve conflicts. Interactive: per-file prompt loop. Non-interactive: if any UPDATE conflict → print diff report, exit non-zero, write nothing.
  9. Print summary. Show what will happen. --dry-run stops here.
  10. Flush. Atomic write of all approved changes.
  11. Update .anvil.lock with new generation context + checksums.
  12. Release the PID lock and exit.
