Mental model

Four jobs, four places

A junior engineer should avoid reading this PR as one giant command. Read it as four small services inside the CLI: entrypoint, context builder, scaffold engine, and post-scaffold runner.

  • cli.ts parses user intent.
  • init.ts orders the workflow and owns safety gates.
  • init-context.ts turns environment facts into a typed ScaffoldContext.
  • engine.ts previews/writes; init-post.ts runs optional setup.

Safety choices

The "boring" guardrails are the architecture

  • Interactive work requires a TTY. If the user pipes into the command without --non-interactive, init exits cleanly before prompts or writes.
  • Dry-run resolves a path but does not create it. The command uses path.resolve for dry-run and only calls mkdir/realpath on real writes.
  • Real writes are locked. Non-dry-run acquires a directory lock and releases it in finally, so failures do not strand the lock.
  • Interrupted lockfiles are explicit. Interactive users can resume; non-interactive runs stop and tell the operator what to do.
Mentor note

Why dependency injection shows up everywhere

This PR makes the command testable by injecting cwd, stdin, prompts, lock acquisition, filesystem probes, command runners, fetch, scaffold, and preview functions. That can look noisy at first, but it lets tests prove important behavior without shelling out or writing real projects.

As a rule: inject the boundary, not the business rule. The business rule still lives in plain TypeScript; only the outside-world effects are replaceable in tests.

Failure design

How the PR handles imperfect environments

  • Latest-version network calls use an abortable timeout and wrap both initial fetch and response body parsing.
  • Network failures produce warnings and fall back to bundled toolchain defaults from Anvil.
  • .git stat failures warn and skip hooks instead of failing the scaffold.
  • Package installs, git init, pre-commit, and Python plugin install are best-effort commands.
  • Cross-language re-scaffold is a hard stop because applying a different manifest over an old lockfile is unsafe.

How to extend it

A junior-friendly change recipe

  1. Adding a new generated file? Add a manifest entry and either a static file or template. Then add manifest/render tests. Do not special-case it in init.ts.
  2. Adding a new prompt/default? Put the decision in init-context.ts, add it to the answer/context data, and test interactive and non-interactive behavior.
  3. Adding a post-scaffold command? Keep it best-effort unless the scaffold cannot be valid without it. Return warnings with manual remediation commands.
  4. Changing dry-run behavior? First prove what must not happen: no directory creation, no lock acquisition, no scaffold flush, no post commands.
  5. Changing a safety branch? Add focused tests and make sure the mutation target still covers the branch. This project treats mutation as a gate.
