1 · Mental model

Think of the plugin as a router

ESLint does not know about individual files by itself. It loads one plugin object, then uses the rule names in that object to find the matching rule module. That is why the smallest reliable design is: plugin.js owns the public names, and each rule module owns one behavior.

Architecture topology static scaffold → generated project → ESLint runtime

Mentor tip: when you inherit a plugin like this, do not start inside the most complex rule. Start at plugin.js. Confirm the public names, then open the matching file. That catches path and module-format bugs before you reason about AST logic.

2 · Call flow

What happens when ESLint runs?

Every rule follows the same lifecycle: ESLint loads the plugin, creates the rule with a context, walks the parsed AST, and lets the rule call context.report() when it sees an anti-pattern.

Sequence diagram load → create → visit → report

Rule shape

Each file exports { meta, create }. meta documents the rule; create(context) returns the AST visitors.

Visitor shape

Rules stay focused: catch-based rules visit CatchClause, comment rules visit Program, branch rules visit CatchClause and IfStatement.

Report shape

Rules report by messageId. Tests assert those IDs so wording can change less often than behavior.

3 · Data flow

How source code becomes a lint finding

The rule does not read raw TypeScript text in most cases. ESLint parses the file first, then sends structured nodes to each visitor. The only exception in this batch is no-placeholder-comments, which intentionally reads the comment stream.

Data flow diagram source text → AST/comments → decision → report

Mentor tip: AST rules are mostly data-classification problems. Write down the shape you expect, write a RuleTester case for it, then make the visitor just smart enough to distinguish that shape from nearby valid code.

5 · Design decisions

Why the code is shaped this way

The implementation is intentionally plain CommonJS and test-first. That keeps the generated scaffold easy to inspect, and it matches the way ESLint flat config imports a local plugin from a generated project.

CommonJS package boundary

The generated TS project may be ESM, but the local ESLint plugin stays CommonJS via the nearest tools/lint-rules/package.json. This prevents module-format drift.

Shared AST helper

ast-utils.js centralizes logging-call recognition and expression unwrapping. That avoids three rules independently deciding what logger.error() means.

Scoped branch logic

RULE-05 only checks catch bodies and error-handling if branches. That keeps it focused on duplicate error reporting instead of every ordinary throw.

Tests as executable spec

Every rule has valid and invalid examples. The tests document boundary behavior, not just happy-path detection.

The pattern every rule follows
module.exports = {
  meta: {
    type: 'problem',
    docs: { description: '...', recommended: true },
    messages: { messageId: 'Human-readable lint message.' },
    schema: [],
  },
  create(context) {
    return {
      CatchClause(node) {
        if (/* rule-specific decision */) {
          context.report({ node, messageId: 'messageId' });
        }
      },
    };
  },
};
6 · How it was built

The engineering workflow behind the PR

This ticket was written in a test-first loop. The important part for a junior engineer is that the tests were not just “coverage”; they were a design tool that caught overscoped and underscoped rule behavior.

1

Start with the public contract

Define the plugin rule names and the expected RuleTester examples from the ticket and spec.

2

Make the tests fail for the right reason

Rule modules were initially missing, so the red state proved the tests were wired to real files.

3

Implement narrow AST visitors

Each rule got just enough traversal to detect its anti-pattern while preserving valid nearby code.

4

Review for false positives and false negatives

Independent review drove regressions for guarded logging, scoped error branches, and conditional default returns.

5

Validate the whole repo

Targeted lint-rule tests, full Bun tests, typecheck, agent check, fixture checks, slop-scan, and PR checks all ran cleanly.

7 · How to extend this next

Adding the next ESLint rule without getting lost

Use the same recipe for TIX-000030 and later rule tickets. The shape should feel boring: tests first, one rule module, plugin registration, then a review pass for overbreadth.

1. Write the examples

  • At least three valid cases
  • At least three invalid cases
  • One boundary case from reviewer intuition

2. Keep the rule local

  • One public rule name
  • One messageId per finding type
  • Shared helpers only when repeated

3. Validate the contract

  • Targeted RuleTester command
  • Full Bun test suite
  • Plugin registration test
